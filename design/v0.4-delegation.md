# v0.4 Delegation — Design Proposal

> **Status:** DRAFT. Written 2026-05-21 after v0.3 feature-complete.
> Captures the architectural thinking before implementation begins.
> Revisable; open questions at §10.

---

## 1. Why now / motivation

v0.3 ships federation: Bob can call `peer.ask` on Alice's daimon if Bob's DID is pinned in Alice's address book with `"peer.ask"` in `ApprovedVerbs`. This is coarse-grained authorization. It answers "who can call what" but not:

- **How much?** No rate limiting or call-count cap.
- **For how long?** No time-bounded access.
- **Under what constraints?** No model restriction, content policy, or spend ceiling.
- **Delegated to whom?** If Alice wants her sub-agent (a smaller LLM running locally) to call Bob's `peer.ask` on her behalf, there is no mechanism.

These missing features block the next milestone: **agent-to-agent commerce**. For a labor market to function (v0.5), an agent must be able to delegate fine-grained authority to another agent that then exercises it independently — without the delegating agent being online or trusted to not exceed the grant.

**Biscuit tokens** solve this. They are cryptographically signed capability tokens with offline-attenuatable Datalog policies, verifiable by anyone who knows the root public key. Because every Daimon already has an Ed25519 keypair (its DID key), Biscuit slots in with no new key infrastructure.

---

## 2. Goals

1. Let a daimon issue a **capability token** granting another agent the right to exercise specific verbs on its behalf.
2. Let token holders **attenuate** (restrict further) the token without contacting the issuer.
3. Let serving daimons **verify tokens offline** — no round-trip to the issuer required.
4. **Compose with v0.3 address book**, not replace it: tokens are an optional upgrade path; pinned-DID authorization remains valid and doesn't require tokens.
5. Ship a **reputation primitive**: a signed receipt chain that proves a daimon has reliably served peer.ask calls — first building block toward the v0.5 labor market.

---

## 3. What is Biscuit?

[Biscuit](https://www.biscuitsec.org/) is a capability token format built by CleverCloud. Key properties relevant to Daimon:

| Property | Relevance |
|---|---|
| **Ed25519-signed root** | Daimon's existing identity keypair is the root authority. Zero new key management. |
| **Offline attenuation** | Token holder can create a sub-token without contacting the issuer. Sub-agent gets a restricted copy. |
| **Datalog policy language** | Arbitrary constraints: time windows, call counts, model allowlists, spend ceilings. |
| **No online verification** | Serving daimon verifies against the root public key embedded in the token. Works air-gapped. |
| **Apache 2.0** | License compatible with Daimon. |
| **Go library** | `github.com/biscuit-auth/biscuit-go` — maintained, Ed25519 support. |

Biscuit v3 is current. The token format is a protobuf-encoded chain of signed blocks. Each block adds facts and checks; a Datalog evaluator confirms the check set passes before the token is accepted.

### Comparison to alternatives

| | Biscuit | JWT (HMAC) | Macaroon |
|---|---|---|---|
| Offline attenuation | ✅ Ed25519 block chain | ❌ need shared secret | ✅ HMAC chain |
| Datalog policies | ✅ | ❌ claims only | ❌ caveats only |
| Ed25519 root | ✅ | ❌ | ❌ |
| Existing Go tooling | ✅ | ✅ | ⚠️ limited |

Biscuit wins on offline attenuation + Ed25519 + Datalog. Those three properties together match Daimon's architecture exactly.

## 4. Capability model

### 4.1 Root capabilities

A root Biscuit token asserts one or more **capabilities** — facts in the Datalog sense. Daimon-specific capability vocabulary (v0.4):

```
// Grant the right to invoke a verb on a specific daimon
right("peer.ask", "did:key:z6MkAlice...");

// Grant the right to invoke peer.ask on any daimon in Alice's address book
right("peer.ask", "any");

// Grant wallet signing rights (future; v0.4.x)
// right("wallet.sign", "evm:base-sepolia");
```

Tokens also encode mandatory checks (constraints that MUST hold for the token to be valid):

```
// Token expires at Unix timestamp 1780000000
check if time($t), $t <= 2026-12-31T00:00:00Z;

// Only valid for a specific model
check if peer_ask_model($m), $m == "claude-haiku-4-5";

// Spend ceiling: max $5.00 in smallest USDC units
check if peer_ask_spend($s), $s <= 5000000;
```

### 4.2 Attenuation by token holders

Alice issues a root token to Bob's sub-agent. Bob can further restrict it:

```
// Bob's sub-agent restricts to today only (already valid-until is 2026-12-31)
check if time($t), $t <= 2026-05-22T00:00:00Z;
```

The new block is signed by the Biscuit append key (a fresh ephemeral key — Biscuit's standard attenuation mechanism). The root key does NOT re-sign. The chain is: `[root_block, attenuated_block]`.

Bob's sub-agent cannot expand the grant (cannot remove Alice's "expires 2026-12-31" check; cannot add rights Alice didn't give). Offline attenuation is always in the direction of restriction.

### 4.3 What a token does NOT grant

- Network access to Alice's daimon socket. Tokens are authorization artifacts, not authentication credentials. The Noise IK handshake still handles transport authentication.
- Bypass of the address book entirely. In v0.4, a token is checked AFTER the address book's "is this peer connected?" gate. The token gates the specific verb invocation.

## 5. Token lifecycle

### 5.1 Issuance — `daimon.capability.issue`

```
daimon.capability.issue({
  verbs: ["peer.ask"],        // which verbs to grant
  grantee_did?: string,       // optional: restrict to a specific peer DID
  valid_until?: string,       // RFC3339; default: now + 24h
  constraints?: object,       // Datalog facts to embed as checks
  max_calls?: number,         // call count ceiling (applied as Datalog check)
}) → {
  token: string,              // base64url-encoded Biscuit token
  token_id: string,           // UUID for revocation and audit
  expires_at: string,         // RFC3339
}
```

The token is signed with the daimon's Ed25519 private key (the DID key). The root block contains the `right(...)` facts for the requested verbs.

Audit row: `capability.issued { token_id, verbs, grantee_did?, valid_until, constraints? }`.

### 5.2 Attenuation — `daimon.capability.attenuate`

```
daimon.capability.attenuate({
  token: string,              // input token to attenuate
  constraints: object,        // additional Datalog checks to embed
}) → {
  token: string,              // new attenuated Biscuit token
}
```

This is a local operation — no network call to the issuer. The output token is strictly more restrictive than the input. Implemented using `biscuit-go`'s append operation.

No audit row on the issuing daimon (can't know about offline attenuations).

### 5.3 Verification — serving daimon on inbound peer.ask

When a peer presents a Biscuit token alongside a `peer.ask` call, the serving daimon:

1. Decodes the token's root public key (from the first block's signature).
2. Looks up the corresponding DID: `identity.DIDFromPublicKey(rootPubKey)`.
3. Checks that the DID is in the local address book (or on a trust allowlist for token-bearing peers). This confirms the token is from a recognized issuer.
4. Runs the Biscuit verifier: does the token authorize `right("peer.ask", serving_daimon_did)`?
5. Checks all embedded constraints (time, model, spend ceiling) against the current context.
6. If all checks pass: serves the call. If any fail: returns `CodeCapabilityDenied` (`-32014`).

Wire change for `peer.ask` in v0.4:

```json
{
  "jsonrpc": "2.0",
  "method": "peer.ask",
  "params": {
    "provider": "claude",
    "request": { ... },
    "capability_token": "<base64url biscuit>"
  },
  "id": "..."
}
```

`capability_token` is optional in v0.4: if absent, the existing address-book-pinned-DID gate applies. If present, the token is verified first and overrides the pinned-verbs check for the verified verbs.

### 5.4 Revocation — `daimon.capability.revoke`

```
daimon.capability.revoke({
  token_id: string,
}) → {}
```

Adds `token_id` to the local revocation list (SQLite table in `$DAIMON_HOME/capability.db`). The serving daimon checks this list during verification step 3. Revocation is local-only — cannot retroactively revoke tokens held by others — but the issuer's serving daimon will refuse to honour them.

Audit row: `capability.revoked { token_id }`.

### 5.5 Listing — `daimon.capability.list`

```
daimon.capability.list({
  include_revoked?: bool,
}) → {
  tokens: [{ token_id, verbs, grantee_did?, valid_until, revoked: bool, issued_at }]
}
```

Lists tokens THIS daimon issued. Does not include tokens issued by others (those are untrackable after offline attenuation).

## 6. Integration with v0.3 address book

The address book remains the primary authorization primitive. Capability tokens are an opt-in upgrade:

| Scenario | Authorization mechanism |
|---|---|
| Bob's DID is pinned with `peer.ask` | Address book gate (existing v0.3) |
| Bob presents a valid Biscuit token | Token verification (v0.4) — bypasses need to be pinned |
| Bob presents a token AND is pinned | Token is checked first; pinning is redundant but harmless |
| Bob is `blocked` in the address book | Token is rejected even if valid — block takes absolute precedence |

In `dispatchPeer`, the check order becomes:
1. Is the peer `blocked`? → reject (always, tokens cannot override a block)
2. Is a capability token present in the params? → verify it → if valid, serve the verb
3. Is the peer pinned with the verb in `ApprovedVerbs`? → serve the verb
4. None of the above → `CodePeerUnauthorized`

## 8. Reputation primitive

### 8.1 What reputation means in Daimon

A daimon's reputation is evidence that it has reliably served peer verbs in the past. Evidence takes the form of **signed receipts** — compact signed statements issued by the serving daimon after a successful call.

### 8.2 Receipt format

After a successful `peer.ask` call, the serving daimon can issue a receipt:

```json
{
  "version": 1,
  "receipt_id": "<uuid>",
  "served_at": "<rfc3339>",
  "served_verb": "peer.ask",
  "caller_did": "did:key:z6MkBob...",
  "server_did": "did:key:z6MkAlice...",
  "provider": "claude",
  "model": "claude-opus-4-5",
  "input_tokens": 14,
  "output_tokens": 23,
  "duration_ms": 1240
}
```

Signed with the serving daimon's Ed25519 key. The receipt is returned to the caller alongside the normal `peer.ask` response when `"request_receipt": true` is in the params.

### 8.3 Receipt accumulation

The calling daimon collects receipts from every daimon it has successfully transacted with. A daimon's reputation is the set of receipts it can present — each signed by a different serving daimon's key (verifiable via their DID).

```
daimon.reputation.receipts() → {
  receipts: [{ receipt_id, served_at, server_did, served_verb, ... }]
}
```

### 8.4 Receipt verification

Any third party (another daimon considering whether to accept a peer.ask call) can verify:
1. Parse `receipt.server_did` → extract Ed25519 public key.
2. Verify the signature over the receipt JSON.
3. The receipt is genuine: server_did did serve caller_did at the stated time.

This requires no central authority. Receipts are portable, offline-verifiable, and accumulate naturally through normal federation activity.

### 8.5 What reputation gates in v0.4

In v0.4, reputation receipts are informational only — a daimon can show them but no gate checks them automatically. They become a gate input in v0.5 when the labor-market primitive needs to decide which agents to trust with a task bid.

## 9. New error codes (v0.4)

| Code | Constant | Meaning |
|---|---|---|
| `-32014` | `CodeCapabilityDenied` | Biscuit token presented but verification failed (expired, wrong right, revoked, invalid signature). |
| `-32015` | `CodeCapabilityRequired` | The serving daimon requires a capability token for this verb but none was presented. Reserved; not used in v0.4 (both token and pin-based auth are accepted). |

## 10. Non-goals for v0.4

- **Token transfer / secondary market**: tokens are issued 1:1 for a grantee and tracked by `token_id`. Re-gifting a token to a third party is possible at the cryptographic level (Biscuit doesn't prevent it) but the serving daimon can check `grantee_did` if set. True secondary-market token trading is v0.5+.
- **Cross-chain token redemption**: capability tokens don't involve payment. The x402 flow in v0.3 (peer.pay.required + USDC settlement) is the payment primitive; tokens are the authorization primitive. They will compose in v0.5.
- **Reputation score aggregation or global leaderboard**: receipts are verifiable facts; scoring/ranking is a layer above, deferred to v0.5.
- **Token refresh / rotation**: tokens expire and must be re-issued. Automatic refresh is out of scope.
- **did:web or did:ion as root key**: v0.4 uses did:key only, same as v0.3 transport.

## 11. Open questions

1. **Token storage on the caller side**: where does a calling daimon store tokens it receives from others? SQLite table? File? We need a `capability.db` schema for inbound tokens analogous to the address book for DIDs.

2. **Spend tracking for `max_calls` / `max_spend`**: Datalog checks are stateless — they verify a fact presented at verification time. For "max 100 calls" to work, the serving daimon must track call count per token_id and present it as a Datalog fact during verification. This is a stateful check; needs a `calls_used` table in `capability.db`.

3. **Receipt storage**: receipts issued by the serving daimon should be logged (for audit), and receipts received by the calling daimon should be stored (for reputation display). Two separate storage concerns — probably same `capability.db` but separate tables.

4. **Blocking token abuse**: a malicious caller could replay a token up to its expiry. Mitigations: (a) short token lifetimes (default 24h), (b) token_id revocation, (c) `max_calls` cap per token. All three should be recommended in the CLI UX.

5. **Biscuit-go stability**: `github.com/biscuit-auth/biscuit-go` is maintained but not at v1.0. We should pin a specific commit hash and audit for breaking changes rather than `go get latest`. The core Biscuit spec is stable (v3); the Go bindings lag slightly.

6. **Capability discovery**: should `daimon.federation.config` advertise that this daimon accepts capability tokens (vs. pin-only)? A `"capability_tokens": true` field in the config response would let callers know not to bother with address-book pinning flows.
