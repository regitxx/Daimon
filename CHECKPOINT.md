**Seed lifecycle complete (sessions 46–48, 2026-05-19):** the BIP-39 mnemonic now has full round-trip portability between a daimon and the user's external backup / wallet of choice. Three pieces:

- **`daimon wallet show-mnemonic` + `client.wallet.show_mnemonic` / `showMnemonic`** (sessions 46–47) — password-gated re-display. The daemon re-runs the full Argon2id KDF + AES-GCM-decrypt against the on-disk keystore on every call, NOT short-circuiting on in-memory state. Wrong password surfaces as new typed `-32008 CodeWrongPassword`, distinct from `-32001 CodeIdentityLocked` — CLIs MUST NOT rewrite this as "run daimon unlock first." Industry standard for non-custodial seed reveal (MetaMask, Phantom, Trezor, Ledger Live).
- **`daimon wallet recover`** (session 48) — offline counterpart: write a fresh keystore from a user-supplied 12- or 24-word BIP-39 phrase, so an external backup imports into a fresh daimon. CLI-only by design (no SDK wrapper): a live seed swap on a running daimon would orphan every wallet derived from the previous mnemonic. Refuses if a keystore already exists at the target path.
- **SPEC §6.1 Wallet + new SPEC §14.6 ("Backup re-display and seed import")** formalise both verbs with normative requirements for third-party Daimon-compatible implementations.

The lifecycle now covers all three real-world entry points: **generate fresh** (auto-create on first unlock), **see what you generated** (`show-mnemonic`), **bring your own** (`recover`).

The production lifecycle now works end-to-end:

1. `daimon init` resolves `$DAIMON_HOME` (default `os.UserConfigDir()/daimon`), prompts for a password twice with confirmation, generates a fresh Ed25519 identity, and writes the encrypted keystore at mode 0600. Refuses to overwrite without `--force`.
2. `daimon unlock` resolves the same home, prompts for the password once, dials `$DAIMON_HOME/daimon.sock`. If the daemon isn't running, **auto-spawns `daimond serve` as a fully-detached process** (own session via Setsid, stdout/stderr to `$DAIMON_HOME/daimon.log`), polls the socket with bounded backoff up to 5s, then sends `daimon.identity.unlock {password}`. The daemon loads the keystore, opens the persistent memory store and activity log, and transitions from locked to unlocked via an atomic flag with release/acquire semantics. The unlock RPC is the **only** method the dispatcher allows pre-unlock; everything else returns `CodeIdentityLocked`. Idempotent: a second unlock against an already-unlocked daemon returns the same DID without re-deriving the key.
3. `daimon identity get` dials the existing socket and round-trips `daimon.identity.get` — the post-unlock smoke test that proves the gate also permits valid calls. Detects "daemon not running" (ENOENT/ECONNREFUSED) and "daemon locked" (CodeIdentityLocked) and prints actionable hints rather than raw errors.

The same `daimond` binary still runs the self-contained 8-step demonstration via `daimond demo` (ephemeral identity, temp socket, never touches `$DAIMON_HOME`) — the living spec from session 4 is preserved. The CLI and the daemon both route filesystem layout through the same `internal/daimonhome` resolver so they cannot disagree about where the keystore lives or where the socket should be opened (sun_path-overflow fallback to `$TMPDIR/daimon-$uid.sock` lives in that one helper).

## What we are building

**Daimon** — a protocol giving every human one sovereign agent for life. Portable, encrypted, owned by them. Holds their memory, identity, reputation, and money. Plugs into any AI model or service through an open protocol.

Not a chatbot. Not a wrapper. A substrate. The email-standard moment for agents.

In Socratic philosophy, the *daimon* (δαίμων) was your inner guiding voice — uniquely yours. The double meaning with Unix daemon is intentional: your daimon literally runs as a daemon on your machine.

## Why this and not something else

- **No incumbent can build it.** Anthropic, OpenAI, Google profit from lock-in. Cross-provider portable identity cannibalizes their business. The hole is permanent.
- **Composes, doesn't compete.** Sits on top of MCP, A2A, x402, DIDs. Not in the protocol war — above it.
- **Single-player utility from day one.** Even with zero other users: persistent memory across providers, no re-explaining yourself, switch Claude → GPT → local Llama without losing your agent.
- **Network value emerges naturally.** Once you have portable identity + memory + reputation, agent commerce, sub-agent delegation, agent labor markets become trivial.

## v0.1 — what daimon-core does (and only this)

1. **Identity**: did:key (ephemeral) + did:ion (anchored). Keypair in OS keychain. Passkey-authenticated.
2. **Memory**: encrypted, structured long-term store (SQLite + vector index). Versioned, exportable, signed.
3. **Protocol endpoint**: tiny JSON-RPC API any LLM client can call. MCP-compatible.
4. **Provider adapters**: thin shims for Claude, OpenAI, Gemini, Ollama. Calls route through the daimon.
5. **Activity log**: every action signed and logged. Seed of reputation. Exportable. Yours.

37. ~~**Python SDK session 3 — `provider.stream` (the deferred streaming verb).**~~ ✅ shipped 2026-05-11 (session 36). New `sdk/python/daimon/_stream.py` (200 lines): `StreamHandle` class — iterable yielding delta strings, with the terminal envelope (`{response, injected_memory_ids?}`) on `.final` once iteration completes; context-manager support (`__enter__` / `__exit__`) for deterministic socket close on early exit. `open_stream(socket_path, method, params, timeout)` opens the socket, sends the request envelope, returns a started `StreamHandle`. **Architectural decisions:** (1) **Class-based iterator over generator-with-return-value**: Python 3's `yield`/`return` generator pattern hides the terminal envelope in `StopIteration.value`, which is awkward to access. A class with `.final` attribute exposes it naturally without forcing the caller into a `try/except StopIteration` dance. (2) **No half-close on the write side** — unlike `_rpc.py`'s one-shot path where `sock.shutdown(SHUT_WR)` after sending the request lets the server's `json.Decoder` see EOF promptly, streaming keeps the socket bidirectional because the server keeps writing notifications until the stream completes; half-closing here could surface as a write-side error on some kernels. (3) **Newline-delimited frame reader**: each wire frame is one JSON object + `\n` (Go's `json.Encoder.Encode` always appends `\n`). Splitting the recv buffer on raw `\n` bytes is safe because JSON-encoded strings escape literal newlines as `\\n` (two ASCII chars), so the only `\n` byte on the wire is the frame separator. Buffer-based `_read_one_frame` returns one decoded dict per call, accumulating partial reads across `recv` boundaries. (4) **Forward-compat for unknown notification kinds** — the SDK silently skips any notification method that isn't `daimon.provider.stream.delta`, mirroring the Go CLI's `rpcStream` behaviour. Future tool-call deltas, role markers, etc., won't break older SDKs. (5) **Terminal-frame error mapping is identical to non-streaming**: `-32001` lifts to `DaemonLocked`; everything else to `RPCError` via the existing `_from_error_object`. Same exception family, same `.code`/`.message`/`.data` shape. **StubDaemon extension**: new `stream(method, deltas, terminal)` registration alongside `handle`. `deltas` is `list[str]` or a callable `(params) -> list[str]`; `terminal` is a dict (sent as `result`), a `StubRPCError` (sent as `error`), or a callable. The server's `_serve_one` was refactored to read one newline-terminated request (instead of "all bytes until peer half-close"), then dispatch to the streaming or non-streaming handler — both wire shapes now work on the same code path, and all 35 existing non-streaming tests still pass verbatim because the SDK still appends a trailing newline on the request line. **Test count:** 35 → 45 SDK tests (+10 streaming): yields all deltas in order, terminal envelope populates `.final`, assembles nested request from flat kwargs, passes inject_context verbatim, zero-deltas terminal-only case (provider-side fallback-to-invoke), forward-compat skip of unknown notification kinds, terminal RPCError, terminal `-32001` → DaemonLocked, context-manager early-exit closes socket cleanly, peer-closes-without-terminal raises. All race-clean.
