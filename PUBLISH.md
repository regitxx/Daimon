# Publishing the Daimon SDKs

This file is the playbook for cutting a release of `daimon-protocol`
(PyPI) and `@daimon-protocol/sdk` (npm). It's deliberately concrete: every
step is a verbatim command. The Go daemon + CLI are not published as
a package right now — users build from source via `make build`.

> The PyPI distribution name is `daimon-protocol` because the
> unqualified `daimon` name on PyPI is held by an unrelated dormant
> hobby package and `daimon-sdk` / `daimon-client` are both taken by
> active unrelated projects. The import name in user code remains
> `daimon` (`from daimon import Client`). The npm package is scoped
> as `@daimon-protocol/sdk` under the `daimon-protocol` npm organization
> (the bare `daimon` org on npm was already taken, so the org was
> registered under the same disambiguated name as PyPI for symmetry).

The two SDKs are released together in lock-step on shared semver
(`0.1.0`, `0.1.1`, ...). Don't drift them; the live cross-language smoke
tests rely on wire-shape parity.

## Prerequisites — one-time setup

1. **PyPI account + token.** Create at <https://pypi.org/account/register/>,
   then [generate a project-scoped API token](https://pypi.org/manage/account/token/)
   for `daimon-protocol`. (For the very first publish, you'll need an
   account-scoped token; project-scoped tokens can only be issued
   against an already-published project.) Store as the password under
   user `__token__` in
   `~/.pypirc` or pass via `TWINE_USERNAME=__token__ TWINE_PASSWORD=pypi-...`.
2. **npm account + organization.** Create at <https://www.npmjs.com/signup>,
   then `npm login`. The package is scoped (`@daimon-protocol/sdk`) so you
   also need to be a member of the `daimon-protocol` organization on npm.
   Org creation is web-only (<https://www.npmjs.com/org/create>); pick the
   free plan for unlimited public packages. Two-factor on the org is
   recommended; `npm publish` will prompt for the OTP.
3. **Local tools.**
   ```sh
   python -m pip install --upgrade build twine
   (cd sdk/typescript && npm install)
   ```

## Pre-release checklist

Run from a clean checkout of `main`:

1. **CI green on the release commit.** The
   [`ci.yml`](./.github/workflows/ci.yml) workflow runs the full matrix
   (Go + Python 3.10–3.13 + Node 18/20/22) on every push to `main`. The
   release commit must be green before you cut a tag.
2. **Version is bumped.** Both manifests must move in lock-step:
   - [`sdk/python/pyproject.toml`](./sdk/python/pyproject.toml) — `version = "0.1.0"`
   - [`sdk/typescript/package.json`](./sdk/typescript/package.json) — `"version": "0.1.0"`

   PyPI uses PEP 440 (`0.1.0.dev0` → `0.1.0`); npm uses semver
   (`0.1.0-dev.0` → `0.1.0`). Don't try to align the pre-release
   suffixes byte-for-byte — they're constrained by each registry's
   parser. Just align the public version (the leading `0.1.0`).

   **Both runtime version constants are codegen, NOT edited by hand:**
   - Python: `sdk/python/daimon/_version.py` is generated by
     `python scripts/gen_version.py` from `pyproject.toml`'s
     `[project] version`. The CI matrix's Python shard re-runs
     this and asserts `git diff --exit-code daimon/_version.py`,
     so a missed local re-run trips a PR-time failure rather than
     a stale release.
   - TypeScript: `sdk/typescript/src/version.ts` is generated by
     `node scripts/gen-version.mjs` from `package.json`'s
     `version`. Chained ahead of `npm run build`, `npm run
     typecheck`, and `prepublishOnly` so it's regenerated on every
     build automatically.

   Workflow: bump the manifest, run the codegen, commit both
   files together. The Go binary's version is injected at link
   time via `-X main.version=...` (see "Cutting a binary release"
   below) — no source-level constant to bump.
3. **CHANGELOGs are accurate.** Move entries from `[Unreleased]` to a
   new dated section in each SDK's `CHANGELOG.md`. Drop one-liners
   into the README's status line if the public-facing surface changed.
4. **Local smoke.** From the repo root:
   ```sh
   (cd sdk/python && rm -rf dist && python -m build && python -m twine check dist/*)
   (cd sdk/typescript && npm run build && npm pack --dry-run)
   ```
   Twine check must report `PASSED` on both wheel and sdist. `npm pack
   --dry-run` must list `LICENSE`, `CHANGELOG.md`, and `dist/` contents.
   `prepublishOnly` in `package.json` reruns typecheck + test + build
   on actual publish, so a stale dist will be caught — but verifying
   here saves a round-trip with the registry on real publish.
5. **Cross-language streaming smoke.** Optional but recommended for
   `0.x.0` minors: run [`examples/streaming/`](./examples/streaming) end-to-end
   against a real daimon and Ollama. If anything regresses in the
   wire shape, both example scripts will fail in the same way and the
   audit chain verify will mismatch.

## Cut the release

1. **Tag the release commit.** From `main`, on the commit that passed CI:
   ```sh
   git tag -a v0.1.0 -m "Daimon v0.1.0: daemon + CLI + Python/TS SDKs + 4 streaming providers"
   git push origin v0.1.0
   ```
   GitHub auto-renders the tag as a release; you can optionally promote
   it to a full release on the web UI with CHANGELOG copy.

2. **Publish to PyPI:**
   ```sh
   cd sdk/python
   rm -rf dist
   python -m build
   python -m twine check dist/*
   python -m twine upload dist/*           # prompts for token if not in ~/.pypirc
   ```
   Verify at <https://pypi.org/project/daimon-protocol/>.

3. **Publish to npm:**
   ```sh
   cd sdk/typescript
   npm publish                              # respects publishConfig.access: public
   ```
   `prepublishOnly` runs `npm run typecheck && npm run test && npm run
   build` first, so this can't ship a stale dist. Verify at
   <https://www.npmjs.com/package/@daimon-protocol/sdk>.

## Cutting a binary release

The Go daemon + CLI ship as platform tarballs on
[GitHub Releases](https://github.com/regitxx/Daimon/releases) for
users who don't want to install Go. The pipeline is
`.github/workflows/release.yml`, triggered by any `v*` tag push (or
manually via `workflow_dispatch`).

Recipe for a binary release **after** the SDK publishes above are
done (so the SDK pre-release on PyPI/npm matches what the binary
exposes):

1. **Tag the release commit.** Same commit that the SDKs were cut
   from. The release workflow reads the tag, so the tag has to
   exist before the workflow can fire:
   ```sh
   git tag -a v0.2.0-dev.3 -m "v0.2.0-dev.3: pre-built binary distribution"
   git push origin v0.2.0-dev.3
   ```
   The workflow fires on the push, cross-compiles `daimond` +
   `daimon` for {darwin, linux} × {amd64, arm64}, packages each
   tarball with LICENSE + README + QUICKSTART, computes
   `checksums.txt` with SHA-256 of every tarball, and runs
   `gh release create` with the tag annotation as the release body.

2. **The release.yml workflow flags pre-releases automatically.**
   Regex on the SemVer pre-release pattern `vX.Y.Z-…` → the
   release is marked `--prerelease` in GitHub's UI. Plain `vX.Y.Z`
   stays as a stable release. So `v0.2.0-dev.3`, `v0.2.0-rc.1`,
   `v0.2.0-alpha.1` all flag correctly; `v0.2.0` GA does not.

3. **Verify via install.sh.** From a fresh shell after the workflow
   completes:
   ```sh
   TMP=$(mktemp -d)
   DAIMON_INSTALL_PREFIX="$TMP" sh ./install.sh
   "$TMP/daimon" --help | head -1
   "$TMP/daimond" --help 2>&1 | head -1
   ```
   Both should report the new tag. The install-script CI shard
   does the same dance on ubuntu + macOS post-merge.

4. **Binary version is injected at link time.** The release
   workflow's go build invocation includes
   `-ldflags "-X main.version=$tag"`. Locally-built binaries (via
   `make build`) use `git describe --tags --dirty --always`
   instead, so a working-tree build reports e.g.
   `v0.2.0-dev.3-2-gabc1234-dirty`. There is NO source-level
   version constant in `cmd/daimon` or `cmd/daimond` to maintain.

5. **Re-running a tag.** If the workflow fails midway (e.g.
   GitHub Releases create rate limited) or you need to re-build a
   tag, use the workflow dispatch path:
   ```sh
   gh workflow run release.yml -f tag=v0.2.0-dev.3
   ```
   The workflow's create-or-update step is idempotent — re-runs
   clobber existing assets with `--clobber` and re-set the
   prerelease flag.

## Post-release smoke

From a clean shell (not the repo root), to catch packaging issues that
only show up after the registry has the artifact:

```sh
# Python
python -m venv /tmp/daimon-pubsmoke && source /tmp/daimon-pubsmoke/bin/activate
pip install daimon-protocol
python -c "from daimon import Client, StreamHandle; print('imports OK')"

# TypeScript
mkdir /tmp/daimon-npm-smoke && cd /tmp/daimon-npm-smoke
npm init -y >/dev/null
npm install @daimon-protocol/sdk
node -e 'import("@daimon-protocol/sdk").then(m => console.log("imports OK; VERSION=", m.VERSION))'
```

If a running daimon is available, the smoke scripts in
[`examples/streaming/`](./examples/streaming) work against the
published packages with minor import path swaps (drop the `dist/`
relative import and use `@daimon-protocol/sdk` directly).

## Iterating between releases — the dev cycling pattern

The v0.2 development arc shipped four pre-releases between the v0.1
GA (`0.1.0`) and v0.2.0 GA: `0.2.0-dev.0`, `dev.1`, `dev.2`, `dev.3`.
Each represents a meaningful inflection point in the v0.2 surface,
not just a build-pipeline test:

- `dev.0` (2026-05-18) — initial v0.2 surface (wallet + payment SDKs)
- `dev.1` (2026-05-19) — added `show_mnemonic` SDK wrapper
- `dev.2` (2026-05-19, same day) — added `derive` SDK wrapper
- `dev.3` (2026-05-20) — first pre-release with platform binaries on
  GitHub Releases (no SDK surface changes; binary distribution is the
  new artifact)

The rhythm: bump the dev counter every time the SDK / CLI gains a
verb a user might call on the pre-release channel BEFORE the next
GA. Users on `pip install --pre daimon-protocol` /
`npm install @daimon-protocol/sdk@dev` always get the latest dev.N.
GA cuts when the gate (live integration, in v0.2's case 40.4) clears.

Mechanics:

- Python: `0.1.1.dev0` → `0.1.1.dev1` → `0.1.1.rc0` → `0.1.1`
- TypeScript: `0.1.1-dev.0` → `0.1.1-dev.1` → `0.1.1-rc.0` → `0.1.1`

`npm publish --tag dev` keeps a dev version off the `latest` channel
so `npm install @daimon-protocol/sdk` doesn't pick it up. PyPI
doesn't have a direct equivalent; `pip install daimon-protocol`
always resolves to the highest stable version unless you pass
`--pre`. **Default install paths always resolve to the latest GA**:
this is non-negotiable and is what lets us publish dev pre-releases
freely without affecting v0.1.0 GA users.

Workflow per dev cut:

1. Bump versions in both `pyproject.toml` + `package.json` (the
   codegen scripts pick up the change on next build).
2. Move the `[Unreleased]` block in each SDK's CHANGELOG to a dated
   pre-release block (`[0.2.0-dev.N] — YYYY-MM-DD`).
3. SDK suites green (`pytest -q`, `npm test`).
4. Publish: `python -m build && python -m twine upload dist/*` and
   `npm publish --tag dev`.
5. `git tag v0.2.0-dev.N -m "..."` + `git push origin v0.2.0-dev.N`
   to trigger the binary release workflow (optional per dev cut —
   binary releases need only happen when the daemon's surface
   changed AND a re-distribution is meaningful; SDK-only dev cuts
   can skip the tag).

## If you need to unpublish

PyPI allows file deletion but **not** re-upload of the same version
number — once `0.1.0` is yanked, you must move to `0.1.1`. npm allows
unpublish within 72 hours of publish, after which you must deprecate
instead. Both registries treat unpublish as user-hostile; prefer
shipping `0.1.1` with a fix over yanking `0.1.0`.

Yank PyPI: there's no `twine` subcommand for this — yanks are an
operator action through the web UI at
<https://pypi.org/manage/project/daimon-protocol/releases/>, under
the broken release's "Options" menu → "Yank". Yanked versions stay
installable by pinned-version requests (`pip install daimon-protocol==0.1.0`)
but stop appearing as default resolutions. Document the reason in the
yank form so it surfaces in the resolver error message users see.

Unpublish npm (within 72h):

```sh
npm unpublish @daimon-protocol/sdk@0.1.0
```

Deprecate npm (any time, preferred for older releases):

```sh
npm deprecate @daimon-protocol/sdk@0.1.0 "Use 0.1.1+; 0.1.0 had <issue>"
```
