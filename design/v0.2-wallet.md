# Daimon v0.2 — agent wallet + x402 payments

**Status:** DRAFT — session 1 of a multi-session arc. Nothing here is committed; the strategic shape of v0.2 is huckgod's call. This document exists to put the landscape in one place and surface the open design questions before any code lands.

**Date:** 2026-05-12
**Predecessor:** [SPEC.md](../SPEC.md) (v0.1 protocol)
**Companion:** [JOURNAL.md](../JOURNAL.md) (build log)

## 1. Why this is in v0.2

v0.1 gives the principal *one daimon* that holds their identity, their memory, and their AI provider routing. v0.2's question is: **can that daimon also hold and spend money on the principal's behalf?**

The minute the answer is yes, two things become possible that nothing else in the agent stack solves:

1. **The principal's agent can transact with other agents and with APIs.** Pay-per-call inference, pay-per-document data access, pay-per-task delegation. No subscription, no API keys, no account-creation funnel between agent and service.
2. **The audit log gains a complete financial record signed by the principal.** Every dollar the daimon spent, who it paid, what was returned, all chain-verifiable under the same Ed25519 key that signs every memory row.

x402 is the only payment protocol designed for this access pattern. Composable with HTTP, no out-of-band onboarding, stablecoin-settled, agent-native. It's what v0.2 should build against.

## 2. What x402 actually is (concise)

The HTTP status code `402 Payment Required` has been reserved since 1997 with no canonical implementation. Coinbase shipped x402 in October 2025 as the canonical one. v2 of the spec (early 2026) moved payment data to headers.

**The flow** (from [docs.x402.org](https://docs.x402.org)):

1. Client calls a paid endpoint.
2. Server returns `402 Payment Required` with a `PAYMENT-REQUIRED` header carrying a Base64-encoded `PaymentRequirements` object: which networks/schemes/tokens are accepted, the price, the destination address.
3. Client picks a `PaymentRequirement` it can satisfy, builds a `PaymentPayload` (signed payment authorization), and retries the request with the payload Base64-encoded in a `PAYMENT-SIGNATURE` header.
4. Server (typically via a facilitator) calls `/verify` then `/settle` on the facilitator, gets confirmation, and serves the resource. The settlement outcome rides back to the client in a `PAYMENT-RESPONSE` header.

**Roles:**

| Role | Who |
|---|---|
| Payer | Anyone holding the right token on the right chain. **For v0.2: the daimon.** |
| Resource server | The thing being purchased — a paid API, a paid endpoint, an agent's task interface. |
| Facilitator | Bridges HTTP to chain. Offers `/verify` and `/settle`. Coinbase runs a public one; self-hosted is supported. |

**Supported chains (today):** EVM (Base / Ethereum mainnet / Polygon / etc.), Solana (SVM), Stellar.
**Supported schemes:** `exact` (pay a specified amount). `upto` and others reserved for future.

The protocol is **stateless on the server side**: every call carries its own payment authorization in headers. No accounts, no API keys, no balance kept on the server.

## 3. Where the wallet lives in Daimon

**Proposal:** alongside identity, in `$DAIMON_HOME/`, encrypted under a key derived from the same passphrase that unlocks identity.

```
$DAIMON_HOME/
├── identity.keystore     ← v0.1: Ed25519 identity key, AES-256-GCM
├── memory.db             ← v0.1: encrypted memory rows
├── activity.log          ← v0.1: signed activity log
├── wallet.keystore       ← v0.2: keypair(s) per chain, AES-256-GCM
└── daimon.sock           ← v0.1: RPC socket
```

The wallet keystore is opened on `daimon unlock` alongside identity. Unlock is a single password, single transition; v0.2 doesn't add a second authentication step.

**Key design — separate keys, not the identity key:**

The Ed25519 identity key (`did:key`) is the principal's *identity*. The wallet keys are the principal's *spending authority*. Compromising one shouldn't compromise the other, and chain-specific wallets (EVM secp256k1, Solana Ed25519-on-curve25519, Stellar Ed25519) have different signing requirements anyway. The wallet keystore holds N keypairs — one per chain — derived from a single HD seed or generated independently. **Open question 1:** HD derivation vs independent generation. HD (BIP-32 / SLIP-10) is more compact and well-understood; independent generation is simpler. Both are defensible.

**Open question 2:** does the principal hold *their own* wallet keys, or does v0.2 also support a custodial mode where Coinbase/Circle/etc. hold the keys and the daimon authenticates via OAuth? Non-custodial is the protocol-purity answer; custodial is the get-grandmother-onboarded answer. Probably both, but pick one for v0.2 and defer the other.

## 4. New SPEC sections (proposed)

These slot into SPEC.md as `### N.M`-style additions; everything in v0.1 stays.

### 4.1 Wallet primitive (new §X)

| Field | Type | Notes |
|---|---|---|
| `chain` | string | `"evm:base"`, `"evm:base-sepolia"`, `"svm:solana"`, `"stellar"` (per x402 v2's network identifiers) |
| `address` | string | The public address on that chain |
| `pubkey` | bytes | Raw public key (for callers that need it, e.g. signature verification) |
| `created_at` | int64 | unix ms |
| (no privkey field exposed via RPC — it stays in keystore) |

The wallet keystore is one file, holding N entries (one per chain). On unlock the daemon decrypts the keystore and holds the keys in memory; on lock or daemon kill they're zeroed.

### 4.2 RPC verbs (new §6.1.X)

```
daimon.wallet.list                 → [{chain, address, pubkey, created_at}, ...]
daimon.wallet.create {chain}       → {chain, address, pubkey, created_at}
daimon.wallet.address {chain}      → {address}            (convenience, single-chain getter)
daimon.wallet.balance {chain}      → {balance: {token: ...}, as_of: ms}
                                     ← daemon may proxy via facilitator or direct RPC
daimon.wallet.sign {chain, message_hex}
                                   → {signature_hex}      (raw signing for arbitrary use)
daimon.payment.required {url, method?, headers?}
                                   → {requirements: PaymentRequirements[]}
                                     ← issues a probe request that expects a 402, parses
daimon.payment.pay {url, method?, body?, requirement, ...}
                                   → {response_status, response_body, payment_response}
                                     ← end-to-end: builds PaymentPayload, retries call,
                                       returns the resource response + payment outcome
```

**Composition note:** `daimon.payment.pay` is the high-level "give me the resource, handle the payment" verb. It's analogous to `daimon.provider.invoke` for a provider — the SDK consumer doesn't have to know about HTTP 402 mechanics. **Open question 3:** should `daimon.provider.invoke` itself trip into a payment flow if the configured provider endpoint returns 402? Probably yes for v0.2 — keeps the SDK surface tight — but it adds complexity (the audit-row for `provider.invoke` would need to embed a `payment.*` linkage).

### 4.3 Activity log kinds (new §8.2 entries)

| Kind | Payload | Logged on |
|---|---|---|
| `wallet.created` | `{chain, address}` | `daimon.wallet.create` |
| `payment.required` | `{url, requirement_summary}` | server returned 402 to a daimon call |
| `payment.signed` | `{url, scheme, network, amount, token, destination, payload_hash}` | daimon authorized payment |
| `payment.settled` | `{url, scheme, network, amount, token, destination, tx_hash, success}` | facilitator confirmed |
| `payment.failed` | `{url, reason, requirement_summary}` | facilitator rejected or settlement failed |

Each row is hash-chained and Ed25519-signed by the *identity* key (not the wallet key — the wallet key signs payment payloads on chain, the identity key signs audit rows in the local log). Same `activity.verify` walk as v0.1; just new kinds in the dispatch table.

### 4.4 Memory record kinds (provisional)

x402 doesn't necessarily need new memory kinds — the audit log already records every payment. But if the principal wants the daimon to *remember* relationships ("this API charges 0.5¢ per call and is reliable", "this agent agreed to do tasks for me at this rate"), that's memory. **Open question 4:** new memory kind `payment.history`? Or fold into `observation`? My weak preference is to NOT add a new kind — observations work, and the audit log is the source of truth for transactions themselves. Memory is for *learned context*, not for raw transaction records.

### 4.5 Provider routing implications (new §7.3)

If a provider's endpoint sits behind 402:

- v0.1's `provider.invoke` flow: POST request → 200 with envelope.
- v0.2's flow: POST request → 402 → daimon builds `PaymentPayload` from wallet → POST retry with `PAYMENT-SIGNATURE` header → 200 with envelope. Audit log gets a `payment.settled` row chained adjacent to the `provider.invoke` row.

The chain row that says `provider.invoke streamed=true matched=0` would gain (when 402-mediated) a `payment_id: 01K...` field pointing at the adjacent `payment.settled` row, so a downstream auditor can reconstruct "this Claude call cost $0.003 settled in USDC on Base." Tight.

Each phase is a session or two. Total v0.2 arc is probably 4–6 sessions of work.

| Phase | Scope | Out of scope |
|---|---|---|
| **40.0 (this doc)** | Design doc, landscape survey, open questions surfaced. | Any code. |

EVM keypair generation (secp256k1) as the first chain. Unit tests at the wallet-package level. | RPC, CLI, SDK. No chain interaction yet. |
| **40.2 — RPC surface** | `daimon.wallet.list / create / address / balance / sign` handlers. Activity-log kind `wallet.created` wired. `cmd/daimon/cmd_wallet.go` for `daimon wallet list / create / address`. Server tests like v0.1's handler tests. | Payments, facilitators. |
| **40.3 — x402 client** | Pure HTTP-402-flow code in `internal/payment/`. Parse `PAYMENT-REQUIRED` Base64 payload. Build `PaymentPayload` for the `exact` scheme on EVM. Sign with the wallet key. POST retry with `PAYMENT-SIGNATURE` header. Parse `PAYMENT-RESPONSE`. Activity-log kinds `payment.required / signed / settled / failed`. | Multi-chain (just EVM `exact` first). Custodial mode. |
| **40.4 — facilitator integration** | Wire to a real facilitator (Coinbase's public one or a self-hosted instance). Live smoke against a real x402-protected test endpoint. | Production-volume usage; that's v0.2.x polish. |

| **40.6 — SDK verbs + docs** | Python + TS SDK wrappers for `wallet.*` and `payment.*`. CHANGELOG + README updates. Cross-language smoke (similar to session 38) buying a small thing from an x402-protected test endpoint. | Anything beyond EVM `exact`. |

After 40.6, v0.2.0 ships and v0.3 design (federation / A2A) opens.

## 6. Open questions (in priority order for huckgod's call)

1. **Custodial vs non-custodial wallet.** Hold own keys (protocol-pure, harder onboarding) vs OAuth-style integration with Coinbase / Circle / etc. (easier onboarding, custodial trust). Pick one for v0.2; defer the other.
2. **HD derivation vs independent keypairs.** Both are defensible. HD (BIP-32 / SLIP-10) is industry standard but adds complexity.
3. **Should `provider.invoke` auto-pay 402 responses?** Yes is more user-friendly; adds complexity to the audit-row chain. No keeps the verbs orthogonal and forces the SDK consumer to explicitly opt into payment.
4. **Memory kind for payment history?** Probably no — observations are sufficient and the audit log is the source of truth — but worth confirming.
5. **First chain.** EVM (Base specifically) is the obvious pick — that's where the x402 reference implementations and most facilitators live today. Stellar is interesting for low-fee small-payment economics. SVM (Solana) is interesting for throughput. My weak vote: EVM first, defer multi-chain to v0.2.x.
6. **Facilitator: public vs self-hosted.** Coinbase's public facilitator is the path of least resistance. Self-hosted lifts the trust assumption. v0.2 should probably make the facilitator endpoint configurable per `daimon.toml` (or env) so neither is privileged in the codebase.
7. **Receive-side semantics.** v0.2 as drafted is a payer-only design — the daimon spends, doesn't earn. If the daimon should also *receive* x402 payments (e.g. another agent calls your daimon's exposed endpoint and pays), that's a server-side flow with very different mechanics. Probably defer to v0.3 (when federation lands and your daimon has a public-facing identity anyway), but worth flagging.

---

## 7. Out of scope for v0.2

- **Agent-to-agent payments where both sides are daimons.** That's a federation question (v0.3): how do two daimons discover each other, negotiate, and settle. v0.2 just gives a single daimon spending power against existing x402 servers.
- **Reputation / credit / IOUs.** v0.4.
- **Token issuance.** The daimon spends existing tokens; it doesn't mint anything.
- **Privacy mechanisms beyond standard x402.** No zk-proofs of payment, no shielded transactions. Same privacy posture as any normal stablecoin transfer.
- **Compliance / KYC / sanctions screening.** Out of protocol scope. Facilitators may impose this on their side; that's between the principal and their chosen facilitator.
