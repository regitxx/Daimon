## Why this document exists

v0.1 is GA. v0.2 is structurally complete (pre-release across all three channels). v0.2.0 GA waits on live Base Sepolia settlement — an external unblock. **The implementation work blocking is over; the next architectural decisions are about v0.3.**

v0.3 = **federation**: daimons that talk to other daimons over the network. The SPEC § 1.2 names three primitives:

1. Public DID resolution and verification by third parties
2. Agent-to-agent communication (the A2A protocol surface)
3. Daimon as payment RECIPIENT (the server side of x402)

…plus an implicit fourth: encrypted transport between daimons (the MLS / Noise composes-with row in the README).

This doc proposes a concrete architecture for all four. The goal is to have something to debate, fix, and refine before code happens — not to commit to it.

## Goals

- **Single-player utility preserved.** v0.3 must not break the v0.1 + v0.2 invariant that a daimon is useful with zero peers. The federation surface should be opt-in, off by default, and degradable: a daimon with networking disabled stays a fully working v0.1+v0.2 daimon.
- **Cryptographic continuity.** A daimon's `did:key` is its identity. Federation MUST use the same Ed25519 keypair the audit chain is signed with — no separate "network identity" with its own key management surface. Whatever transport we pick keys directly off the existing identity.
- **No new trust roots.** Daimon's promise is "you own it." A federation layer that requires trusting a central directory, a third-party CA, or an opaque overlay network breaks that promise. Discovery + authentication must work with only user-owned primitives (DNS-under-domains-you-control, manual pinning, peer-to-peer protocols).
- **Composability over inversion.** v0.3 should compose with existing standards (DID, mTLS, MLS, A2A, x402-recipient) rather than invent parallel ones. The whole project's framing in `README.md` is "sits above the protocol stack" — federation specifically must hold to that.

## Non-goals (deferred to v0.3.x or v0.4+)

- **Group communication / MLS.** v0.3 ships 1:1 daimon-to-daimon channels only. Group conversations (multiple daimons in one encrypted channel) need MLS, which is significantly more complex than Noise IK. Defer to v0.3.x.
- **Public daimon directory / search.** A registry where you can look up "all daimons interested in topic X" requires either a centralized service or a DHT, both of which have their own threat models. v0.3 ships only direct addressing (you already know the peer's DID + endpoint). Discovery as a separate primitive is v0.4 work.
- **Reputation primitives.** v0.4 territory. v0.3 channels are authenticated but unjudged — "is this peer trustworthy" is the user's call, not the protocol's.
- **Mobile daimons.** v0.3 assumes a daemon process on a long-lived machine (or at least one with an addressable endpoint). Mobile / NAT'd / firewalled daimons need a rendezvous service. Useful but separable.
- **Multi-tenant daimons.** A daemon process still represents exactly one principal in v0.3. Multi-tenancy is v1.x.

## Architectural decisions

### Decision 1: Transport = Noise IK over QUIC

**Choice.** Each daimon-to-daimon channel is a [Noise IK](https://noiseprotocol.org/noise.html#interactive-handshake-patterns-fundamental) handshake over [QUIC](https://datatracker.ietf.org/doc/html/rfc9000), with the static keys on both sides being the daimon's `did:key` Ed25519 keys.

**Alternatives considered:**
- **TLS 1.3 with self-signed certs derived from did:key.** Pros: ubiquitous tooling, standard library support. Cons: cert ASN.1 layer adds gratuitous complexity for a 1:1 keyed channel; modern Noise IK gives the same security with a fraction of the wire weight.
- **MLS from day one.** MLS is the right answer for group conversations but for 1:1 channels it's overkill — the tree machinery doesn't pull weight. Save it for v0.3.x.
- **Plain TCP + ChaCha20-Poly1305 (custom framing).** Cleaner wire shape but requires hand-rolling everything Noise already gives us (forward secrecy, mutual auth, replay protection).

**Why Noise IK specifically:** Noise IK is the simplest pattern where both sides have a static key and the initiator already knows the responder's static key — which matches our model exactly (you address a daimon by its DID, which IS its static key). One handshake, two crypto messages, mutual authentication, forward secrecy.

**Why QUIC over plain UDP/TCP:** QUIC gives us multiplexed streams (multiple concurrent RPC calls over one Noise handshake, no head-of-line blocking), connection migration (laptop closes lid, IP changes, channel survives), and low-latency 0-RTT for the reconnect case. The cost is a heavier library dependency (`quic-go` is well-maintained but adds ~50k LoC to the binary); for this primitive, worth it.

### Decision 2: Discovery = DID resolution → endpoint URL

**Choice.** A daimon's DID resolves to a JSON document containing one or more `endpoint` URLs. Resolution happens via the DID method:

- **did:key** (default, offline): the DID itself encodes the pubkey but NOT an endpoint. To talk to a `did:key` daimon, you need its endpoint URL from out-of-band sharing (manual paste, QR code, business card, email). This is the "no new trust roots" path — pure peer-to-peer, no infrastructure.
- **did:web** (new in v0.3): a DID anchored to a domain the user controls. `did:web:example.com:dmn:abc` resolves to `https://example.com/.well-known/did-document/dmn/abc`, which is a JSON document signed by the daimon containing the endpoint URL. Lets users with their own domain advertise their daimon at a stable address without depending on any third party.
- **did:ion** (deferred): the Sidetree-based DID method SPEC § 4.1 already names. Stable across key rotation, but resolution requires ION node access. Reserved for v0.3.x once we know whether the friction of a Bitcoin-anchored identity is worth the rotation property.

The endpoint document also carries the daimon's `transport_pubkey` (the same Ed25519 key the DID encodes — included for clients that pre-Noise resolve), `protocols` (an array of supported A2A verb namespaces), and optional `payment_address` (the on-chain address this daimon receives x402 payments at — see Decision 4).

**Alternatives considered:**
- **DHT discovery.** Distributed hash table where DIDs resolve to endpoints via peer-to-peer lookup. Pros: no centralized service. Cons: a whole new networking subsystem; vulnerable to eclipse attacks; latency. Maybe v0.4.
- **Centralized directory (daimon-registry.org).** Pros: simple, fast lookup. Cons: violates "no new trust roots" — the directory becomes a control point. Hard no.
- **DNS TXT records.** Roughly equivalent to did:web; did:web is more flexible (multiple daimons per domain) and the JSON document carries richer data.

### Decision 3: Wire surface = extend the existing JSON-RPC

**Choice.** Federation verbs live under `daimon.peer.*` and `daimon.federation.*` in the same JSON-RPC 2.0 envelope the local Unix-socket surface uses. New verbs:

- `daimon.peer.dial({did, endpoint?})` — open a Noise IK channel to a peer. Returns a `channel_id` to use in subsequent calls. `endpoint` is optional if the DID resolves to one (did:web / did:ion).
- `daimon.peer.close({channel_id})` — tear down a channel.
- `daimon.peer.list()` — show open channels (channel_id, peer DID, opened_at, last_activity).
- `daimon.peer.invoke({channel_id, method, params})` — invoke a method ON the peer daimon. `method` is whatever A2A protocol verb the peer supports (e.g. `peer.echo`, `peer.ask`, `peer.pay`). The peer's response comes back as the JSON-RPC result.
- `daimon.peer.address_book.{add, list, remove, pin}` — persistent record of "daimons I've talked to," signed into the audit chain. `pin` marks a peer as "I trust this DID — auto-accept future channel opens."

The protocols the SERVING daimon exposes are configured via `daimond serve --protocol peer.echo --protocol peer.ask --protocol peer.pay` (or via a config file). Default: none. **Servers are opt-in to every verb they expose.**

`daimon.federation.config` lets a client introspect what protocols the local daimon serves, what its public endpoint is, what DID methods it speaks.

**Why extend the existing JSON-RPC surface:** keeps the SDK story coherent. A Python or TS client that already calls `client.memory.write({...})` calls `client.peer.dial({...})` the same way. No new transport for the SDK to learn. The daemon-to-daemon link uses the Noise+QUIC tunnel internally, but every client-facing call stays JSON-RPC over the Unix socket.

### Decision 4: Daimon as x402 recipient

**Choice.** A daimon that wants to receive x402 payments configures `payment_address` in its DID endpoint document. The daimon serves a special RPC verb, `peer.pay.required({resource})`, that returns the canonical x402 `PaymentRequirements` (network, asset, recipient, max_amount, validBefore). A paying daimon calls `daimon.payment.pay(...)` against the RECEIVING daimon's endpoint URL exactly as it would against any other x402-protected URL — the v0.2 payment client doesn't need to know whether the resource is a daimon or a generic HTTP server.

**Crucially: settlement still happens on-chain via a facilitator, just like v0.2.** A daimon does NOT itself broadcast transactions; it constructs a valid EIP-3009 signature, the paying side hits the facilitator, the facilitator verifies + settles. The receiving daimon's only job is to advertise the requirement, accept the signed authorization, and verify it landed on-chain (via a `PaymentRecord` confirmation).

**Audit-chain extension:** new activity kinds `peer.payment.invoiced`, `peer.payment.received` so the receiving daimon's chain mirrors the v0.2 paying daimon's `payment.signed` + `payment.settled` pair.

**Alternatives considered:**
- **Receiver runs its own facilitator.** Too much surface area; facilitators are a real product category and we're not building one.
- **Off-chain settlement (state channels).** Doesn't compose with the x402 standard; defer.

### Decision 5: Address book + trust model

**Choice.** A daimon maintains a local `address_book` of peer DIDs it has talked to. Each entry: peer DID, optional pet name, pin status (pinned / first-seen / blocked), per-protocol authorization (which `peer.*` verbs are allowed from this peer), audit log of every call this peer has made.

Trust model:
- **TOFU (Trust On First Use)** for pinned peers — the first time a DID is pinned, the user accepts the (DID, endpoint) tuple. Subsequent connections check the channel's static-key authentication against the pinned DID.
- **Explicit per-verb authorization.** A peer pinned for `peer.echo` cannot invoke `peer.ask` without separate pin. Capability granularity follows the verb namespace.
- **No reputation, no signatures-from-third-parties.** Reputation primitives are v0.4. v0.3 is "I personally vouch for this DID, here's what I let it do."

**Alternatives considered:**
- **Web-of-trust signatures.** Friend-of-friend introductions. v0.4.
- **Allow-by-default.** Too permissive; lets any peer that knows your DID + endpoint open channels.

## Wire surface summary

New RPC verbs (all under the existing JSON-RPC 2.0 envelope, same Unix-socket transport for client-to-daimon, Noise+QUIC for daimon-to-daimon):

| Verb | Direction | Purpose |
|---|---|---|
| `daimon.peer.dial` | client → local daemon | Open a channel to a peer daimon |
| `daimon.peer.close` | client → local daemon | Close a channel |
| `daimon.peer.list` | client → local daemon | Enumerate open channels |
| `daimon.peer.invoke` | client → local daemon → peer | Invoke a method on the peer; result returned to client |
| `daimon.peer.address_book.*` | client → local daemon | Manage the address book (add, list, remove, pin) |
| `daimon.federation.config` | client → local daemon | Introspect what this daimon serves |
| `peer.echo`, `peer.ask`, `peer.pay` | daimon → daimon (over Noise+QUIC) | Example peer-served verbs — minimum v0.3 set |
| `peer.pay.required` | daimon → daimon | Return x402 PaymentRequirements for a resource |

New activity-log kinds:

| Kind | Written when |
|---|---|
| `peer.channel.opened` | Outbound `daimon.peer.dial` succeeds |
| `peer.channel.closed` | Channel teardown |
| `peer.invoke.sent` | This daimon invoked a method on a peer |
| `peer.invoke.received` | A peer invoked a method on this daimon |
| `peer.payment.invoiced` | A peer fetched `peer.pay.required` (about to pay this daimon) |
| `peer.payment.received` | Settlement confirmation for an inbound x402 |
| `peer.address_book.pinned` | User explicitly trusts a peer DID |

New typed RPC error codes:

| Code | Meaning |
|---|---|
| `-32010 CodePeerUnreachable` | Could not establish a channel (DNS/network/handshake failure) |
| `-32011 CodePeerAuthFailed` | Noise handshake completed but the peer's static key didn't match the expected DID |
| `-32012 CodePeerProtocolUnsupported` | Peer doesn't serve the requested method |
| `-32013 CodePeerUnauthorized` | Address book says this peer isn't authorized for this verb |

## Incremental rollout plan

Implementing v0.3 in a single big-bang commit would be brittle. Proposed phasing (one phase per merge boundary):

**Phase 30 — Discovery primitive:**
- did:web resolver in `internal/did/web.go`
- Endpoint document spec, signing, verification
- `daimon.federation.config` introspection verb
- Unit tests against canonical did:web fixtures
- No actual networking yet

**Phase 31 — Noise IK + QUIC transport:**
- `internal/transport/noise_quic.go`
- Channel multiplexing, lifecycle, idle timeout
- `daimon.peer.{dial,close,list}` verbs
- Stub remote-daimon in test harness for handshake testing
- No application-layer verbs yet

**Phase 32 — Address book + TOFU pinning:**
- Persistent address book (on-disk, encrypted under identity key like memory.db)
- `daimon.peer.address_book.*` verbs
- Activity-chain rows for pin/unpin
- Per-verb authorization checks in the dispatcher

**Phase 33 — First peer-served verb (`peer.echo`):**
- Simplest possible A2A protocol: send a message, get it echoed back signed.
- Validates the full stack (dial → handshake → invoke → response → audit).
- Cross-daimon smoke in CI: spawn two daemons on different ports, dial, echo, verify both audit chains.

**Phase 34 — peer.ask (cross-daimon provider.invoke):**
- One daimon proxies its LLM provider through a peer. Effectively "I have OpenAI access, my friend doesn't — let them call my OpenAI through me."
- New audit kind: `peer.invoke.served`. This is the first verb with real economic implications (peer is consuming the local daimon's API credits).
- v0.3.x: rate limit + pay-per-call gating.

**Phase 35 — x402 recipient (peer.pay.required + settlement):**
- `payment_address` in endpoint document
- `peer.pay.required` verb
- Audit chain `peer.payment.{invoiced,received}` rows
- Cross-language smoke: TS-SDK daimon pays a Python-SDK daimon, both audit chains verify

**Phase 36 — SDK wrappers:**
- `client.peer.*` in both Python + TS SDKs
- Cross-language tests against the test harness
- README + QUICKSTART updates

GA criterion: phases 30–36 all green, cross-daimon smoke runs in CI, real-world dogfood between two daimons on huckgod's machines for one full week without manual intervention.

## Open questions (where I'm least sure)

1. **NAT traversal.** A daimon behind a residential NAT can dial out but can't be dialed into. The endpoint URL in did:web assumes a publicly addressable endpoint. We'd need either ICE/STUN (complex), a rendezvous service (centralization), or restrict v0.3 to daimons with public endpoints. **Tentative answer:** restrict v0.3 to public endpoints; ship NAT traversal as v0.3.x via a user-runnable relay (the user runs the relay, doesn't depend on ours).

2. **Endpoint document signing key.** did:key has one key (the identity Ed25519). Should the endpoint document be signed by that same key, or should there be a separate "endpoint signing key" that can be rotated more easily? **Tentative answer:** same key. Adding a separate key adds a key-rotation surface area that we don't need yet; if the identity key is compromised, the daimon is compromised regardless.

3. **`peer.ask` quota model.** If you let a peer invoke your LLM provider, you pay for it. What's the right model — rate limiting, per-call budget, pay-per-call via x402, all of the above? **Tentative answer:** start with rate limit only (N calls/hour per peer DID), add x402 gating in v0.3.x.

4. **Audit chain split — local vs federated.** Should `peer.*` audit rows live in the same activity.log as `memory.*` and `payment.*`, or a separate `peer_activity.log`? **Tentative answer:** same log. The chain integrity property is "every action this principal has authorized, in order"; splitting hides the cross-cutting view.

5. **Compatibility with existing A2A protocol.** The A2A Project at https://a2aproject.dev/ has a published protocol. Should `peer.*` be A2A on the wire, or a Daimon-specific wire format with an A2A adapter? **Tentative answer:** Daimon-specific wire format (Noise+QUIC + JSON-RPC) for the v0.3 baseline; add an A2A adapter as a separate provider-style shim once A2A is mature.

6. **Backwards-compatible with v0.2.0 GA?** If a v0.3 daimon talks to a v0.2 daimon, what happens? **Tentative answer:** v0.2 daimons don't expose any network endpoint; the v0.3 capability is purely additive. A v0.3 client that tries to dial a v0.2 daimon gets `CodePeerUnreachable` (no endpoint). No protocol-version negotiation needed.

7. **Test infrastructure.** The x402-smoke shard exists because we run a mock x402 server in CI. The equivalent for v0.3 would be running two daemons in CI and having them talk. Doable but heavier (two persistent processes, real Noise+QUIC handshake). **Tentative answer:** worth it; the wire-shape parity protection has paid off for v0.2.

## What I want feedback on

If you're reading this with revision authority (or are huckgod), the specific calls I'd love pushback on:

- **Noise IK + QUIC vs alternatives.** Have I picked the right transport? Anything I'm missing on the alternatives front?
- **did:web vs other DID methods.** Is the "user runs their own domain" assumption too strong? Should we ship something easier first (manual endpoint sharing) and add did:web later?
- **Rollout phasing.** Phases 30–36 — too many, too few, wrong order?
- **Non-goals.** Anything I've deferred that should actually be v0.3 scope? (Group / MLS, public directory, mobile?)
- **The whole shape.** Is "v0.3 = federation = Noise channels + did:web + x402 recipient + address book" the right framing, or should v0.3 be smaller (just discovery + dial, no recipient) and the recipient story slip to v0.3.x?
